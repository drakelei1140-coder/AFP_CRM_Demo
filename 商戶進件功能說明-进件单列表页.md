## 商戶進件功能說明

### 商户进件列表页

### 1. 功能定位

商戶進件列表用於集中查看 **商戶（MID）在各通道下的進件記錄**，包括通道側識別信息、可用狀態、進件進度、進件單狀態與錯誤回傳。\
本頁支持在 **不修改商戶主檔資料** 的前提下，針對某一條進件記錄發起 **修改信息後重新遞交進件**（僅作用於該次向通道提交的內容）。

本頁主要服務於：

1.  運營人員按商戶、通道、狀態等維度排查進件問題；
2.  根據通道返回的錯誤信息，在彈窗中調整 **本次遞交給通道的字段值**；
3.  在不改動商戶主檔的前提下，完成進件資料修正、重新遞交與歷史追溯；
4.  與商戶詳情、商戶編輯等頁面分工：**列表僅處理「通道進件」語境下的重試與補資料遞交**，不替代主檔維護。

***

### 2. 數據粒度說明

*   **列表一行 = 一個商戶在一個通道下的一條當前有效進件記錄**
*   同一商戶若對應多個通道，則展示多行；
*   同一通道若發生多次重新進件，列表頁仍只展示 **最新一條當前有效進件記錄**；
*   每一次重新進件，都需要另外記錄一筆 **進件歷史記錄**，供後續查看歷史與時間軸；
*   若與後台主數據命名不一致，以統一字段字典為準。

#### 2.1 首次生成進件記錄的時點

商戶進件記錄的首次生成時點如下：

*   當商戶（MID）滿足進入渠道進件條件時，系統自動生成對應通道的進件記錄；
*   若所選通道 **僅為 AFP**，則在商戶完成基礎資料審核並進入 AFP 對接流程時，自動生成 `Adyen_AFP` 的進件記錄；
*   若所選通道中 **包含任意非 AFP 通道**，則需先完成我方完整審核流程，待允許進件後再生成對應通道的進件記錄；
*   商戶選擇了多個通道時，只要其中 **包含任意非 AFP 通道**，都按完整審核流程處理。

***

### 3. 列表區

#### 3.1 列表顯示字段

列表按表格行展示，建議列順序如下：

| 列名     | 說明                                                                                                |
| ------ | ------------------------------------------------------------------------------------------------- |
| 商戶名稱   | 商戶主檔展示名稱（與商戶詳情一致口徑）。                                                                              |
| 商戶號    | 平台商戶編號（MID）。                                                                                      |
| 通道     | 當前進件所屬通道（與系統內通道主數據一致）。第一期一般只有 `Adyen_AFP`。                                                        |
| 通道商戶號  | 通道返回或分配的商戶標識；尚無則顯示 `-`。                                                                           |
| 通道可用狀態 | 通道側是否可用 / 開通等狀態，以標籤或枚舉文案展示；具體值與通道協議對齊。                                                            |
| 通道進件進度 | 進件流程節點（如：待遞交、處理中、成功、失敗）；進件成功 / 進件失敗的判斷依據見《AFP API 对接完整流程.md》。如果尚未拿到最終成功或失敗結果，統一顯示為【處理中】。          |
| 進件單狀態  | 用於記錄本條進件單在我方系統內的處理狀態，見 3.2。                                                                       |
| 通道返回信息 | AFP 進件時會同時調用多個接口（例如 LE、AH、BL、Store、Payment Method 等），可能返回多個結果信息。列表格子中顯示最新錯誤信息摘要；點擊後彈窗展示各接口返回結果明細。 |
| 更新時間   | 當前進件記錄最後一次更新時間。                                                                                   |
| 操作     | 見 3.3。                                                                                            |

##### 布局要求

1.  各列寬度以可讀為優先；錯誤信息列可適當加寬或支持換行（最多建議 2～3 行後省略）。
2.  允許橫向滾動，不因過窄導致無法辨識主鍵信息。
3.  文本過長時支持懸浮展示完整內容。

#### 3.2 進件單狀態

由於進件資料可能會被 OPS 修改後再次流轉，因此除通道進件進度外，還需要增加一個 **進件單狀態** 字段，用於記錄該條進件單在我方系統內的流轉情況。

建議枚舉如下：

| 進件單狀態 | 說明                                        |
| ----- | ----------------------------------------- |
| 待遞交   | 首次生成後，尚未正式向通道遞交。                          |
| 處理中   | 已向通道發起進件，正在等待通道結果。                        |
| 進件失敗  | 通道已返回失敗結果，可由 OPS 發起修改信息重新進件。              |
| 待风控审核 | 僅適用於包含非 AFP 通道的場景。OPS 修改進件資料後，需要再次進入風控審核。 |
| 待重新進件 | 已完成 OPS 修改，且滿足再次遞交條件，等待系統或操作人重新遞交。        |
| 進件成功  | 通道側進件完成。                                  |

#### 3.3 操作區

每行操作項如下：

| 操作       | 說明                                    |
| -------- | ------------------------------------- |
| 修改信息重新進件 | 僅在【進件單狀態 = 進件失敗】時顯示，打開修改信息彈窗，見第 4 節。  |
| 進件歷史記錄   | 打開彈窗，查看本條進件記錄的歷史遞交記錄、修改記錄與時間軸，見第 6 節。 |

操作列固定於表格最右側；與數據列同一行。多按鈕時縱向排列、頂部對齊。

#### 3.4 操作按鈕顯示規則

*   只有當 **進件單狀態 = 進件失敗** 時，才顯示【修改信息重新進件】按鈕；
*   當進件單狀態為【處理中】【待风控审核】【待重新進件】【進件成功】時，不顯示【修改信息重新進件】按鈕；
*   【進件歷史記錄】按鈕始終可見。

### 4. 修改信息重新進件

#### 4.1 觸發方式

*   點擊某一行的 **【修改信息重新進件】**，彈出 **修改信息彈窗**；
*   本文要求：**不打開商戶編輯頁**，僅在當前列表頁彈窗內完成本次修改。

#### 4.2 面板內容加載

*   自動加載 **上一次向該通道進件時所使用** 的字段名稱與字段值，即與該條進件記錄關聯的遞交快照，而非實時商戶主檔全量；
*   根據不同通道，記錄並展示該通道進件所需的信息；
*   本期 AFP 場景下，需要展示《AFP API 对接完整流程.md》中所涉及接口對應的必要字段，並與我方企業 / 商鋪 / 商戶資料中已有字段取交集；
*   必須滿足傳給 AFP 的所有必填字段；這些字段在我方系統映射的字段也應設為必填；
*   字段以表單形式展示，標籤為字段名稱，值為可編輯控件（文本、下拉、日期、附件等）；
*   若某字段上次為空，則展示為空，允許填寫。

#### 4.3 遞交前處理規則

當 OPS 人員修改資料並點擊確認後，不同通道場景按以下規則處理：

##### 4.3.1 僅 AFP 通道

*   若該進件單的通道 **僅為 AFP**，則 OPS 修改完畢後，**不需要再次進入我方風控審核**；
*   系統可直接把該進件單流轉為 **待重新進件**，並重新向 AFP 發起進件；
*   重新進件後，進件單狀態更新為【處理中】。

##### 4.3.2 包含非 AFP 通道

*   若該進件單的通道中 **包含任意非 AFP 通道**，則 OPS 修改完畢後，**仍需要先進入我方風控審核**；
*   此時進件單狀態先更新為【待风控审核】；
*   風控審核通過後，進件單狀態再更新為【待重新進件】；
*   之後才允許再次向對應通道發起重新進件；
*   重新進件後，進件單狀態更新為【處理中】。

#### 4.4 遞交行為

*   彈窗底部提供 **取消 / 確認** 按鈕；
*   點擊確認後，將 **彈窗內當前值** 作為本次向通道重新進件的 payload；
*   **重要**：通過此入口完成的修改與重新進件，**不得寫回或變更商戶原始主檔資料**；
*   僅更新與 **該通道進件記錄** 相關的遞交內容、進件單狀態、通道進件進度、通道返回信息與歷史記錄。

#### 4.5 結果與反饋

*   若為 AFP 場景，點擊確認後可直接進入重新進件流程；
*   若為非 AFP 場景，點擊確認後先提示已提交風控審核；
*   列表對應行需刷新以下字段：
    *   進件單狀態
    *   通道進件進度
    *   通道返回信息
    *   通道可用狀態
    *   通道商戶號
    *   更新時間
*   失敗時應在彈窗或全局提示中展示錯誤原因。

### 5. 篩選區域

#### 5.1 功能定位

篩選區位於列表上方，用於在進件記錄較多時縮小範圍，快速定位目標商戶或通道。

#### 5.2 交互形式

*   篩選區支持 **展開 / 收起**；
*   默認收起為兩行核心條件，展開後顯示更多；
*   同類列表頁的展開 / 收起規則、按鈕位置與樣式保持一致；
*   桌面端採用與其他列表頁一致的響應式柵格布局；
*   **查詢、重置、展開 / 收起** 的順序與企業 / 商鋪 / 商戶列表頁保持一致。

#### 5.3 篩選字段

建議至少支持以下字段：

| 字段      | 控件類型    | 說明              |
| ------- | ------- | --------------- |
| 商戶名稱關鍵詞 | 輸入框     | 模糊匹配商戶名稱。       |
| 商戶號     | 輸入框     | 精確或模糊匹配 MID。    |
| 通道      | 下拉單選或多選 | 可選值來自通道主數據。     |
| 通道商戶號   | 輸入框     | 支持按通道商戶號檢索。     |
| 通道可用狀態  | 下拉單選    | 枚舉與列表列展示一致。     |
| 通道進件進度  | 下拉單選或多選 | 與進度枚舉一致。        |
| 進件單狀態   | 下拉單選或多選 | 與進件單狀態枚舉一致。     |
| 是否有錯誤返回 | 下拉單選    | 是 / 否。          |
| 更新時間    | 日期範圍    | 按進件記錄或最後遞交時間篩選。 |

#### 5.4 操作按鈕

*   **查詢**（demo 中不需要做此按鈕交互）
*   **重置**（demo 中不需要做此按鈕交互）
*   **展開 / 收起**

### 6. 進件歷史記錄

#### 6.1 觸發方式

*   點擊某一行的 **【進件歷史記錄】**，打開歷史記錄彈窗。

#### 6.2 展示內容

歷史記錄彈窗中，需展示該條進件單的歷史數據與時間軸，包括但不限於：

1.  每次遞交時間
2.  每次遞交操作人
3.  每次遞交前的進件單狀態
4.  每次遞交後的進件單狀態
5.  每次遞交的通道進件進度結果
6.  每次遞交返回的錯誤摘要
7.  每次修改過的字段內容
8.  OPS 修改時間與修改人
9.  風控審核時間與審核人（如有）
10. 最終成功 / 失敗結果

#### 6.3 展示形式

*   建議使用 **時間軸 + 表格摘要** 的形式展示；
*   時間軸用於展示節點事件；
*   表格用於展示每次遞交的字段變更與結果摘要；
*   歷史記錄僅供查看，不在該彈窗內再次編輯。

***

### 7. 與其他頁面的關係要求

1.  **商戶詳情 / 商戶編輯**：主檔資料變更仍走原有編輯與審核流程；本列表中的修改信息重新進件 **不視為編輯主檔**。
2.  **通道主數據**：列表中的「通道」展示需與系統內通道配置一致。
3.  **商戶審核流程**：
    *   若所選通道僅為 AFP，則可跳過我方風控審核後直接進入 AFP 進件流程；
    *   若所選通道中包含任意非 AFP 通道，則仍需走完整審核流程。
4.  **歷史追溯**：本頁為進件歷史與重試的主要查看入口，不要求在商戶主檔頁面中重複承載完整歷史。
